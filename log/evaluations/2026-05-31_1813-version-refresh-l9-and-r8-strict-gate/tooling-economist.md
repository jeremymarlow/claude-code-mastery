That is exactly the right tool (git stash as a cheap time-machine) used to avoid the expensive failure mode — conflating an inherited bug with one's own work and rabbit-holing. _Evidence:_ "does check-strict fail on the COMMITTED state too? ## What worked / what didn't (both parties)

**Claude — strengths.** This is a near-textbook maintenance session for tool economy. Claude treated the repo's own `make` suite as the source of truth, grounded every assertion in a tool result, batched independent probes, and used `git` as an investigative instrument (the `-S` pickaxe to find the regressing commit, stash to isolate inherited failures). It showed real discrimination about *which* tool and *which* values to touch — leaving per-key provenance alone, matching JSON encoding to keep diffs minimal, placing strict-gating at the right workflow layer. Its option-menu communication (R8 fix A/B/C with a recommendation and a rejected-with-reason) is exactly the right weight: enough structure to decide, no plan-mode overhead.

**Claude — weaknesses.** The repeated Edit-before-Read misfires are the clearest economy leak — small, but the same avoidable error three times. The JSON-twin encoding detour, while recovered, points at a tooling gap (no twin-regeneration target) that Claude could have proposed hardening since it had just hit the sharp edge. And while it surfaced the pre-commit-hook asymmetry well, it stopped short of recommending the one change (hook → strict) that would have *mechanically* prevented the exact regression it spent a third of the session diagnosing — leaving that leverage on the table for the human to pick up.

**Human — strengths.** The mid-stream interrupt to inject the 2.1.158/2.1.159 history was the session's pivotal move and a model of when-to-interject fluency; it converted a wrong trajectory into a clean one before any rework landed. The asks were well-sequenced (fix → review → refresh → investigate → harden), each small enough to keep turns economical, and the "also update the skill or CLAUDE.md to use strict" instruction showed the author thinking about *durable* tooling fixes, not just the immediate symptom — recognizing the close-unit command as the right home from memory ("I believe there's one already for make check").

**Human — weaknesses.** Little to fault. The only thing left implicit was a decision on the pre-commit-hook strictness question Claude raised; the session ended with that leverage unresolved (deferred to capture rather than answered), so the mechanical backstop that would prevent recurrence is still non-strict.

## Bottom line

A high-economy maintenance session: a deterministic, check-suite-anchored refresh executed with right-sized tools, well-batched probes, and genuine judgment about which values and which enforcement layer to touch — capped by a model piece of git archaeology (pickaxe + stash) to root-cause and prove a *pre-existing* strict-gate regression. The human's perfectly-timed context injection steered it off a defensible-but-wrong path before any rework, and the fix landed where it durably belongs (the close-unit command and CLAUDE.md). The only economy leaks are minor and repeated — Edit-before-Read misfires and a JSON-encoding detour — plus one piece of leverage left on the table: Claude diagnosed that a non-strict pre-commit hook let the regression through, yet recommended strict only at the human-driven "done" points, not at the hook that would catch it mechanically.
