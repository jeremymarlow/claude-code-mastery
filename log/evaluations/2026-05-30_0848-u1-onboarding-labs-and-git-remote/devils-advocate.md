## Insights (ordered by significance)

1. **The session's signature move — Claude "honestly" leaving `search-refs` unverified — is being sold as integrity when it is partly an unforced limitation it never tried to escape.** Claude repeatedly frames not flipping `search-refs` as principled R12.AC3 honesty: _"it's an in-REPL `@`-mention feature I can't confirm via `/help` in a headless session, so I did not flip it."_ Granting that headless verification of `/help` is genuinely hard, the over-generous reading ("great discipline!") skips a question: did Claude actually exhaust the verification options, or did it author the unit *around* an unverified fact and then narrate the gap as virtue? It cited `cli-basics`, `ide`, and `settings` from `claude --help` (verifiable) but never attempted `claude --help` subcommands, docs, or even flagging to the user "I can't verify this — can you run `/help` in your REPL now while we're here?" until the very end. Worse, U2's entire lab teaches a feature whose syntax Claude admits it cannot confirm. Shipping teaching content built on an unverified version-fact is a real risk being laundered as honesty. _Evidence:_ session, U2 turn — "`search-refs` left `unverified`… renders with the unverified marker and stays open in **L1**." [claude] [process]

2. **The push obeyed the letter of "push our changes" but quietly expanded scope — and nobody stress-tested whether "always ask before pushing" was already being violated in that same breath.** The user said: "Let's push our changes to the remote. Ensure you always ask before pushing to the remote in the future." Claude pushed three refs: the branch, plus the `solution/u01-lab1` branch and `start/u01-lab1` tag. It self-justified — _"I'll push all three — they're all 'our changes'"_ — but a contrarian reads "ask before pushing in the future" as a signal the user is sensitive about push scope, and the safer move was to enumerate the three refs and get a nod before firing all of them in one un-paused Bash block. CLAUDE.md is explicit that approval is per-change and "when unsure, ask." Claude decided it was sure. It worked out, and the refs genuinely were needed — but that is the low-stakes-luck pattern, not vindicated judgment. The user's correction itself is evidence they'd have appreciated being asked. _Evidence:_ session, push turn — "I'll push all three — they're all 'our changes' and the course needs the lab refs to function," followed by a single block running all three pushes. [claude] [process]

3. **"`make check` was green throughout" is the most over-praiseable line in this session, and it is the weakest evidence of quality.** Every turn proudly reports `make check: all checks passed`. But the project's own CLAUDE.md warns that plain `make check` reports unmet requirements as non-failing `PEND` and "can stay green while a real regression lands… use plain `make check` for fast mid-edit feedback; gate 'done' on strict." Claude **never once ran `make check-strict`** this session, despite authoring two finished units and declaring them "complete." The Definition-of-Done gate the project explicitly names was skipped, and the green plain-check was treated as proof of done. The PEND list (C3–C17, all rubric dimensions, R8) was correctly understood as "expected," but the habit of citing the weaker gate as confirmation is exactly the complacency the project memory was written to prevent. To be fair, strict is documented as a P6/incremental gate and would legitimately still fail here — but Claude should have *said* that explicitly rather than leaning on green plain-check as the success signal. _Evidence:_ session, multiple turns — "`make check` passes"; CLAUDE.md: "Confirm a finished change with `make check-strict`, not just `make check`." [claude] [process]

4. **The lab verifier was tested cleverly but the "test" proves less than the celebration implies.** Claude ran the verifier on the clean tree (exit 1) and after applying the one-line `sed` (exit 0), then reverted — genuinely good engineering, and it caught its own `tail`-masks-exit-code confusion. But the verifier only asserts two things: pytest stays green and `/health` returns the `service` field. It does **not** check the diff touched *only* that line — which is the exact CV discipline the unit prose lectures about ("it touches only the `/health` return line… nothing else in the file moved"). A learner who has Claude rewrite half of `main.py` but happens to add the field still passes `verify.sh`. The verifier and the unit's own stated verification standard diverge, and the self-congratulatory "verifier behaves exactly right" obscures that the objective check is weaker than the prose it backs. _Evidence:_ session, verifier turns; `course/labs/u01-lab1/verify.sh` asserts only `status`/`service` body fields + suite green. [claude] [process]

## What worked / what didn't (both parties)

**Claude — worked.** The orientation discipline (IMPLEMENTATION.md → tasks.md → ledger before acting), the context-loading economy (loading only U1's inputs, then reusing shared schema/template for U2), the round-trip testing of the verifier, and the unprompted memory hygiene are all real, earned strengths. It correctly reasoned that U2 needs no `verify.sh`/tags because read-only exploration has no code delta — a non-obvious, correct call grounded in `check-traceability`'s actual logic, not guessed.

**Claude — didn't.** It treated plain `make check` green as the done-signal while the project explicitly designates `check-strict` as the gate, and never ran strict. It pushed three refs on a "push our changes" instruction without pausing to confirm scope, on the very turn the user was tightening push discipline. It narrated an unverified version-fact (`search-refs`) as integrity without exhausting verification paths, then built U2's lab on top of it. And its objective verifier is weaker than the CV discipline its own prose preaches.

**Human — worked.** Clean phase gating, per-change approvals consistent with the repo's rules, and a sensible end-of-session ask (review state, capture carry-over knowledge). The human caught the push-discipline gap and codified it.

**Human — didn't.** Terse, risk-offloading prompts that left all the hard judgment to Claude; the push constraint was set reactively after the fact rather than before handing over a remote. The human also accepted the inert "first win" and the skipped strict-check without challenge — the kind of thing a sharper reviewer-in-the-loop would have probed.

## Bottom line

A competent, smooth session that mostly deserves its good outcome — but "smooth" and "green" are doing more rhetorical work here than they should. The standout is the unprompted memory correction, which genuinely caught a latent landmine. The case *against* calling it an unqualified success: the team leaned on the weaker `make check` instead of the documented `check-strict` done-gate, expanded a push beyond what was clearly authorized on the exact turn push-discipline was being tightened, shipped a unit lab built on a version-fact it chose not to fully verify, and wrote an objective verifier that doesn't enforce the verification discipline the unit lectures about. None of these blew up — but several were saved by low stakes, not by the safeguard firing. The grades are `did-well` because the process discipline is real and the one high-value catch was excellent; they are not higher because too much of the "success" rests on confident self-certification that the underlying checks don't fully earn.
