## What worked / what didn't (both parties)

**Claude — process (did-well).** This is a near-exemplary build session. Claude oriented itself per the project's read-order protocol, loaded only the design sections and tooling it needed, set up a five-item task plan that mirrored the P4 task breakdown, and built in dependency order (config → models → db → services → api → main) with a smoke test *before* writing the full suite. When the smoke test failed (no-such-table), it diagnosed the actual cause — in-memory SQLite gives each connection a fresh DB and it had skipped lifespan startup — rather than papering over it. It treated the spec's DoD as a checklist of verifiable claims and proved each one. The LOC-vs-target handling is a model of how to treat a soft requirement: don't pad, don't silently miss, surface it.

**Claude — communication (did-well).** The running narration is consistently informative and honest: it explains *why* before each move ("login should be 401, not 403"), reports failures plainly, and the closing summary is well-structured — what landed, the two judgment calls broken out for attention, explicit "nothing is committed," and a concrete next-step pointer to P5/L7. The commit message is thorough and accurate. No overclaiming: the under-target LOC and the single environment warning (starlette deprecation, correctly attributed to the installed package, not its own code) are both named rather than buried.

**Human — process (did-well).** The human's contribution is structural: a spec-driven repo with a self-describing state ledger, plus disciplined milestone gating (start → review-via-summary → commit). The "don't commit without go-ahead" agreement was respected and the human exercised it. The terseness is efficient precisely because the upstream investment in the spec makes it sufficient.

**Human — communication (did-okay).** Three-word instructions are unambiguous given the context, so nothing went wrong. But the human accepted the two flagged judgment calls (LOC under target, the extra `Comment` entity) by silence — "commit P4" implicitly ratified them without comment. That's a reasonable trust posture, but an explicit "fine to ship under-target" would have closed the P4-loc decision more cleanly rather than leaving it as a unilateral call the human waved through. The communication did its job; it just did the minimum.

## Bottom line

A high-functioning build session where the spec-driven scaffolding and Claude's empirical-verification discipline reinforced each other. Claude built ~50 files of working, layered code and a believable messy legacy CLI, proved every Definition-of-Done claim against running code (including reproducing all three seeded bugs), caught and fixed its own errors before the human saw them, surfaced its two spec deviations for review, and kept the state ledger honest — all on three terse human instructions. The through-line: when the spec is rich and the agent treats "done" as something to demonstrate rather than assert, terse human steering scales.
