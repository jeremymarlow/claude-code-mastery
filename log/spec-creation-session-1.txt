  ---
  Q1 — Deliverable: what the finished course physically is

  This is the biggest fork because it determines what we actually build in the
  implementation phase.

  Option A — Markdown curriculum + hands-on labs (my recommendation)
  A self-contained Git repo. Each module is a folder with a lesson.md (the
  teaching), a lab/ directory containing a real sample codebase the learner
  clones and modifies, and a solution.md reference answer. The learner does
  things — runs Claude Code against actual code, builds a hook, writes a
  subagent — rather than just reading about them.
  - Pros: Skills genuinely transfer; learning-by-doing; objectively checkable
  ("did the test pass?"). Best fit for a terminal-native tool.
  - Cons: Most work to author and maintain (we have to build and version a
  sample codebase, write and verify every lab + solution).
  - Best when: the goal is real competence, not just awareness.

  Option B — Markdown lessons only
  Reading-oriented modules with embedded examples and command walkthroughs, but
  no separate runnable lab harness. The learner reads, and optionally tries
  commands on their own projects.
  - Pros: Fastest to author; easiest to keep current; light to maintain.
  - Cons: Weaker skill transfer; "I read about subagents" ≠ "I can build one."
  Harder to prove mastery objectively.
  - Best when: you want a reference/handbook more than a bootcamp.

  Option C — Video/workshop scripts
  Instructor-facing scripts, talking points, and slide outlines for live or
  recorded delivery. Built for a presenter, not a solo reader.
  - Pros: Great for live teaching, high engagement in a room.
  - Cons: Useless to a solo self-paced learner without the instructor; video
  production is out of scope here; scripts go stale fastest.
  - Best when: the primary delivery is a live class or a recorded video series.

  ---
  Q2 — Spec methodology: how we structure this spec

  You used "requirements phase / design phase" language, which is the vocabulary
  of formal spec-driven development. Confirming which flavor.

  Option A — Kiro-style (requirements / design / tasks + EARS) (my 
  recommendation)
  Three files with approval gates between them:
  - requirements.md — user stories + acceptance criteria in EARS notation (a
  disciplined format: "WHEN [trigger] THE SYSTEM SHALL [behavior]", "IF 
  [condition] THEN THE SYSTEM SHALL…"). Every requirement gets a stable ID.
  - design.md — architecture, structure, decisions, traced back to requirement
  IDs.
  - tasks.md — an ordered implementation checklist, each task traced to a design
  component.
  - Pros: Maximum rigor and traceability; nothing gets built on unreviewed
  assumptions; and it doubles as a worked teaching example inside the course
  (the course teaches spec-driven dev, so the course's own spec becomes Exhibit
  A).
  - Cons: More ceremony; EARS feels stiff at first.
  - Best when: you want the spec itself to be exemplary — which fits, since
  we're teaching this.

  Option B — Lightweight PRD + design doc
  One requirements/PRD document plus a design doc, in normal prose — no EARS,
  looser IDs.
  - Pros: Faster, less formal, easier to skim.
  - Cons: Looser traceability; easier for requirements to drift from what gets
  built.
  - Best when: speed matters more than rigor.

  Option C — You decide / hybrid
  I pick the structure that best fits "curriculum-as-product" and justify it
  (likely Kiro-style for the requirements rigor but with a lighter tasks file).

  ---
  Q3 — Mastery proof: how the course validates "elite"

  Determines the core acceptance criteria around assessment.

  Option A — Per-module checks + capstone (my recommendation)
  Every module ends with a Definition-of-Done checklist of objectively 
  verifiable items (a command runs, a file exists, a test passes, an artifact
  matches a rubric). The course ends with a realistic capstone project graded
  against a rubric tied to the capability ladder.
  - Pros: Continuous feedback (learner always knows if they're ready to move on)
  plus a final end-to-end proof. Works for solo learners without a grader.
  - Cons: Most assessment artifacts to design.
  - Best when: you want defensible evidence of mastery at every step.

  Option B — Capstone only
  Modules are purely instructional; one big final project is the sole
  assessment.
  - Pros: Less to build; the capstone is realistic and motivating.
  - Cons: A learner can drift for many modules before discovering a gap; no
  early signal.

  Option C — Self-assessment rubrics
  No graded artifacts at all; each module gives the learner a rubric to score
  their own work.
  - Pros: Lightest weight; no "grading" machinery.
  - Cons: Weakest proof; relies entirely on learner honesty/skill to
  self-evaluate — risky for someone new to the domain who may not know what
  "good" looks like yet.

  ---
  Q4 — Audience mode: who consumes it

  Shapes tone and which files exist.

  Option A — Self-paced solo (primary) + facilitator overlay (my recommendation)
  The core path is a developer working alone, fully self-explanatory. Separately
  filed facilitator notes (pacing, discussion prompts, schedule, grading guide)
  sit on top so the same material can also run as a cohort — without those
  notes ever cluttering the solo path.
  - Pros: Covers both your stated audience now and team training later; one
  source of truth.
  - Cons: A bit more to write (the overlay).

  Option B — Self-paced solo only
  Just the solo path. No instructor materials whatsoever.
  - Pros: Simplest, tightest scope.
  - Cons: If you ever want to teach a group, you'd retrofit facilitator material
  later.

  Option C — Instructor-led cohort
  Built around a facilitator teaching a group: timed sessions, discussion
  prompts, facilitator notes are central, and the content assumes an instructor
  is present.
  - Pros: Best for live group training.
  - Cons: A solo learner can't fully use it alone — contradicts "a seasoned dev
  brings themselves up to speed."

● Agreed — that's a real flaw. "Novice/intermediate/elite" ranks the person, and
  any label below the top reads as a verdict on Sam's worth, which is exactly
  the alienation you're describing. The fix is to stop grading the learner and
  instead describe the capability or the kind of work — progress framed as
  territory covered, not a rung you've been sorted onto.

  Here are four ways to track competency without ranking the person, roughly in
  order of how much I like them:

  Option A — "Can-do" capability statements (no levels at all)

  Borrow the idea behind CEFR's descriptors but drop the A1/B2 labels. Progress
  is a checklist of outcome-phrased statements the learner ticks off:
  - "I can run a multi-file change end-to-end and verify the result myself."
  - "I can build a custom subagent for a recurring job and know when not to."

  The statements describe the work, never the person. Orientation comes from
  "here's what I can now do / what's still ahead," and the capstone rubric
  grades the work product against these statements — not the learner. This is
  the purest fix: there's literally no label to feel bad about.

  - Pro: zero ranking, maximal transferability, doubles directly as the rubric.
  - Con: a flat 30-item checklist can feel shapeless; people like some grouping.

  Option B — Neutral stage names describing the relationship with the tool

  If we do group the can-do statements, name the groups after the nature of the 
  work, not the learner's skill. Something like:
  First Wins → Daily Driver → Force Multiplier → Autonomy & Scale.
  These describe how you're using Claude Code (occasional wins → everyday tool →
  multiplying your output → trusting it with autonomous/parallel work). None of
  them implies you were "lesser" before — "Daily Driver" isn't an insult the
  way "intermediate" is.

  - Pro: keeps orientation/shape, avoids person-ranking.
  - Con: still stages, so some sense of "earlier/later" — but tied to scope of
  work, not competence verdicts.

  Option C — "Territory / map" framing

  Phases named as areas of practice you've explored ("Foundations of context,"
  "Extending the tool," "Automation," "Working at scale"). It's a map of
  coverage, not a rank.

  - Pro: neutral, descriptive.
  - Con: closer to a plain table of contents — weaker as a competency signal.

  Option D — Self-anchored progress only

  No fixed scale; the learner defines their own baseline at the start and
  measures against their past self. Progress is purely "what can I do now that I
  couldn't before."

  - Pro: impossible to feel ranked.
  - Con: no shared yardstick — weakens the capstone rubric (R8 needs a stable
  anchor), and senior devs often want an external bar to clear.

  ---
  My recommendation: A as the foundation, with B's neutral names as light 
  grouping. That is: the real tracking mechanism is can-do capability statements
  (the durable, non-judgmental unit of progress and the rubric's backbone), and
  we cluster them under work-relationship stage names (First Wins / Daily
  Driver / Force Multiplier / Autonomy & Scale) purely for shape. The learner is
  never labeled — only their growing set of capabilities is described.

  This keeps everything R1 was load-bearing for: units still map to a cluster,
  and the capstone rubric (R8) grades the work against the can-do statements
  rather than stamping the person "competent" or "elite." It even lets us retire
  the word "elite" from the learner-facing text entirely — "elite practitioner"
  stays in the spec as our design target, but Sam just sees an expanding list
  of things they can now do.
