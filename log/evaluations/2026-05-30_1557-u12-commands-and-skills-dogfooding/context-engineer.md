## What worked / what didn't (both parties)

**The human** ran this session like someone who understands that the bottleneck is context, not typing. Every prompt either primed (start with §3 status), scoped the next read (`first check the .claude/ directory… plus the unit template`), or pressure-tested an artifact against a continuity standard ("Humans will never really be writing new units, you will"). The standout move was identifying frequent restarts as the real felt need and asking for a session-bootstrap skill — and then insisting on authoring U13 in a *new* session precisely "so we completely dogfood your new skills." That is testing continuity-across-the-boundary the only honest way. The only thing I'd note is that the human's excellent challenges generated three sequential U12 worked-example swaps (unit-check→prime-context, new-unit→close-unit, then a close-unit fix), each requiring a full rework-and-recommit of an already-landed unit; that churn was the right call given U13 wasn't authored yet, but it lived on the edge of "should have settled the artifact design before committing U12 the first time." Claude even named that cost honestly each time.

**Claude** was strong on exactly the axes I judge. It primed before acting, scoped reads tightly, externalized state at the boundary (including correcting stale memory files rather than trusting them), and avoided redundant re-reads. Its self-diagnosis of the Edit-vs-Write error surface and its refusal to invent status are precisely the habits that keep a long multi-session build from drifting. The weaknesses are minor: independent Reads occasionally serialized where one batched block would have done, and the artifact it built first (`unit-check`) had to be discarded — though it was the human, not a context failure, that surfaced that. Communication was clear, honestly costed, and consistently pointed the next session at *files* (the read-order skill) rather than at conversation memory.

## Bottom line

This session is what good context engineering looks like in a long, multi-session build: prime from the live status file, read only what the next step needs, write state down at every boundary (and fix the memory pointer when it drifts), and — most tellingly — turn the session's own continuity protocol into a dogfooded `prime-context` skill so the *next* session reads files instead of recollection. The human drove it with continuity-minded prompts and a deliberate "test it in a fresh session" plan; Claude executed with scoped reading, externalized state, and an honest grasp of its own error surfaces. The only real faults are cosmetic — some un-batched Reads and three rounds of worked-example churn — against a session that otherwise models the discipline this lens exists to reward.
