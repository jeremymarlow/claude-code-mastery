## Insights (ordered by significance)

1. **The functional lens was executed with near-ideal tool economy — scripted batches, not serial probes.** Claude turned the most expensive lens (actually running all seven lab verifiers in both directions) into two shell loops rather than fourteen separate Bash turns. Phase A ran every verifier on `main` with per-lab logging in a single call (`for lab in u01-lab1 u03-lab1 …; do … done`), and Phase B looped the `solution/` checkouts the same way, restoring `main` at the end (_Evidence:_ the "Phase A (fail-clean on `main`)" and "Phase B (pass on `solution/` branches)" Bash blocks). It also pre-read two representative verifiers first to confirm they hit the working tree (not a branch) *before* scripting the sweep — cheap reconnaissance that prevented a wrong-shaped batch. This is exactly the right-sized use of Bash for a deterministic, repeatable check, and it kept the turn count low while covering the deepest "does it work" surface. [claude] [process]

2. **Tracking discipline was good, but the nine `TaskCreate` calls were serialized when they were independent — the one clear batching miss.** Claude set up the full eight-lens plan plus a report task as nine consecutive `TaskCreate` turns, each waiting on the prior result, when none depended on another and they could have been issued together (_Evidence:_ the run of `Task #1 … Task #9 created successfully`). The harness explicitly rewards batching independent calls; this is the textbook case for it. Minor in cost terms, but it is the session's most concrete example of leaving turn-economy on the table. There was also a stray `ToolSearch` (`select:TaskCreate`) immediately before — a small wasted lookup for a tool already in hand. [claude] [process]

3. **Reaching for a build-step + drift-gate to close the `{{vd:}}` gap was the right-sized capability, and grounded in an existing repo pattern rather than invented.** When the raw-token rendering defect surfaced (109 tokens across 14 units), Claude did not hand-patch prose or over-build a generic templating engine. It recognized `progress-checklist.md` as a committed-generated artifact with a `--check` drift gate, then built `render-units` to mirror that exact pattern (`unit.src.md` authored → `unit.md` generated+committed, `--check` into `make check`, `--write` into `make render`) (_Evidence:_ "There's a useful precedent in the repo: `progress-checklist.md` is a **committed generated artifact**…"; the `tools/render-units` Write and Makefile edits). It also read `_common.py`, `parse_frontmatter`, and the consumers of `can_do`/`coverage_areas` *before* touching anything, so the new tool slotted into the existing checker contract without breaking it. This is capability used to its potential: the cheapest mechanism that solves the class of problem, validated against the machine layer first. [claude] [process]

4. **Model fit was reasonable but not scrutinized — Opus 4.8 ran the whole session, including stretches a lighter model would have handled.** The session legitimately warranted Opus for the judgment-heavy work: designing the eight-lens framework, diagnosing that the M4 garble was an R15 graceful-degradation defect, and the P7 methodology call (no new requirements → design gate only). But large mechanical spans — running verifier loops, grepping ID-leakage counts, scripted commit staging — are sonnet-or-lighter work. Neither party raised model choice, and the session closed at only 30% context, so there was no pressure forcing the question. In a course that itself teaches "opus vs. sonnet vs. haiku for the task at hand," a missed opportunity here is the absence of subagent offload (see #5), which is where the real model-economy lever sat. [claude] [process]

5. **The biggest missed leverage: the U2–U16 rollout was scoped as serial in-context work when it is the canonical subagent/headless job.** By session end the plan for Stage C is to roll fifteen units one-by-one in the main thread ("each unit is: `git mv` → apply M1/M2/L1/L3 → `render-units` → eyeball → `make check`"), with U2 as a solo calibration (_Evidence:_ the closing "Stage C — the U2–U16 rollout" turn and "Let's do U2 as a solo-calibration check first"). This is precisely the repetitive, well-specified, parallelizable task that subagents exist for — and the course teaches subagents (U13) and parallel agents via worktrees (U16, W9). Once U1 established the approved voice and the render-and-read procedure was written into `P7-quality-pass.md`, dispatching per-unit (or per-stage-batch) Task agents against that procedure would have been the economical play, with the main thread reserved for the judgment step (reviewing rendered output). Plan mode was also never used despite this being a plan-then-execute shape; the human's "catalog all, fix nothing" + explicit gates achieved the same restraint manually, but the native feature went unused. To be fair, the human's "U2 solo first" instinct is sound calibration — but neither party named the parallel-agent option even as a deferred Stage-C accelerant. [claude|human] [process]

6. **Claude's own cleanup of test residue shows good tooling hygiene — it owned its footprint rather than misreporting it as a product defect.** After Phase B left `taskflow_app/__pycache__/*.pyc` on `main`, Claude investigated with `git ls-files` before concluding, correctly diagnosed it as gitignored bytecode surviving its own `solution/u09` checkout (not a product bug), and removed it to restore a pristine tree (_Evidence:_ "it's my own residue. ## What worked / what didn't (both parties)

**Claude — strengths.** Tool *selection* was consistently right-sized: Grep/Bash for the quantitative ID-leakage and token counts, scripted loops for the functional sweep, a generated-artifact + drift-gate for the systemic rendering defect, AskUserQuestion at real forks, and clean git plumbing for the commit recovery. It reliably read the machine contract (schema, parser, consumers) *before* editing, which is what kept `make check` green through a structural change to how units are stored. It owned its own residue. And it self-corrected its earlier "Lens 8 ✅" verdict when the vd garble proved it wrong, re-classifying it as the most material finding — intellectual honesty that improved the work.

**Claude — weaknesses.** Three economy gaps: the nine serialized `TaskCreate` calls (plus a stray `ToolSearch`); no consideration of model fit or subagent offload for the large mechanical spans; and — the standout — scoping the 15-unit rollout as serial main-thread work when it is the archetypal subagent/parallel-agent job the course itself teaches. There were also a few small self-inflicted rework loops (the stray `>` in the source comment, the `{{vd:key}}` literal in a comment that tripped the token scanner, two "file modified since read" re-reads on `decisions.md`) — none costly, but they nudged the turn count up.

**Human — strengths.** The human ran the economy of the session deliberately: chose "catalog all, fix nothing" to keep the systematic view intact, sharpened the brief with a concrete learner complaint (terseness + ID confusion) that became the spine of the findings, and twice stopped Claude from a wrong tool move with a better question ("must it be at the top?"). Most importantly, the human surfaced both process risks — commit hygiene and lost spec discipline — and insisted on fixing them before proceeding, which is exactly the right place to spend a turn.

**Human — weaknesses.** Few, and minor. The human never asked about parallelizing the rollout or about model choice, so that leverage stayed dormant. The free-text answers that overflowed the AskUserQuestion options were efficient communication but signaled that a couple of those structured prompts were heavier than the moment needed — a shared, not solely human, calibration point.

## Bottom line

A genuinely strong tooling session: the right capability was reached for at almost every fork — scripted batches for the functional run, a drift-gated generated artifact for the systemic rendering defect, structured questions at real judgment points, and clean branch-and-slice git recovery — and the machine contract was always read before it was touched. The economy faults are real but bounded: nine serialized task-creates, no attention to model fit, and the one piece of dormant leverage that matters — a 15-unit rollout teed up as serial main-thread work when subagents and worktrees (which this very course teaches) were the economical instrument. The human earned much of the session's quality by steering tool use away from wrong moves and forcing the methodology back on track before more work piled up. The through-line: capability used well and economically in the small, with the biggest efficiency win left on the table for the next session to claim.
