## The longitudinal arc — capability fluency rose, turn economy never converged

Across 23 sessions the story splits cleanly along the two halves of my lens. **Capability *selection*** — reaching for the right feature at the right fork — was strong from the start and grew genuinely fluent. **Capability *economy*** — batched turns, model fit, no rework — never stabilized; it oscillated session-to-session, regressed hard under environmental stress, and was still leaking the *same* avoidable faults in the final week.

The selection arc is real and upward. Early on, `AskUserQuestion` was already used well at genuine gates (`2026-05-29_1845`, `2026-05-29_2146` with preview mockups). The mid-build sessions added a powerful habit: **verify version-specifics against the live CLI rather than memory** — `claude --help`, `claude agents --help`, registering an MCP server to `✓ Connected` (`2026-05-30_1703`), `ToolSearch`→`WebFetch` with redirect recovery (`2026-05-31_1353`). The peak of selection fluency comes late: a `claude-code-guide` subagent dispatched for the one version-specific unknown (`2026-05-31_1730`), git pickaxe + stash to root-cause a *pre-existing* regression (`2026-05-31_1813`), the committed-tool + thin-skill architecture for transcript capture (`2026-05-31_0832`), and Claude pricing a fan-out design with `du`/`wc` math instead of guessing (`2026-05-31_1908`). The design sophistication of the *instruments* Claude built and chose rose monotonically.

The economy arc has no such trend. The very first session set the template for the recurring failure — two parallel-batch cancellations from coupling fragile commands to load-bearing writes, plus harness-flagged redundant Reads (`2026-05-29_1845`). The single best economy session is `2026-05-30_0725` (build-the-guard-first, heredoc-batched verification). The worst dips are `2026-05-30_0942` (the self-inflicted "flaky console" delusion, dozens of wasted turns, lesson re-broken one turn later) and `2026-05-31_1538` (a ~30-call batch with a `git push` welded in, triggering an API-error cancellation cascade of hundreds of lines). Critically, these are not an early-then-fixed phenomenon: the duplicate-batch / write-read-race disease recurred in `2026-05-31_1629` and `2026-05-31_1730` — the penultimate week — **with the corrective rule sitting unused in Claude's own saved memory the whole time**. That is why I call Claude's economy trajectory `mixed`, not `improving`: selection matured, economy plateaued at "did-okay" and relapsed under pressure.

The human trajectory is `improving`. Gate discipline was solid throughout (per-change commit/push approval, honored in nearly every session), but the human's *tooling-economy interventions* sharpened over time: from ratifying recommendations (`2026-05-29_1845`) to surgically diagnosing root causes — the "flaky console" hypothesis (`2026-05-30_0942`), the write-read race naming (`2026-05-31_1629`), the mid-stream context injection that saved a wrong refresh trajectory (`2026-05-31_1813`), and the token-load probe that forced a measured cost analysis (`2026-05-31_1908`). The human became the session's economy regulator.

## Recurring patterns (ordered by significance)

1. **Tool *selection* consistently outran tool *economy* — the through-line of every session.** Claude reached for the right feature at decision points but leaked turns in execution: serialized independent calls, redundant Reads, Edit-before-Read misfires. This split appears in nearly every leaf, stated explicitly as "capability used well in the small, economy left on the table" or "tool choice near-flawless, turn economy the weak axis." _Evidence:_ `2026-05-31_1730` ("tool choice was right-sized... tool economy leaked"), `2026-05-30_1316` ("capability use strong where it counted... left leverage unused at the edges"), `2026-05-30_0816`, `2026-05-31_1235`. [claude]

2. **Duplicate-batch / write-read-race waste — a chronic, self-inflicted, memory-documented relapse.** Firing the same Read/Bash multiple times in one batch, or coupling a write and its dependent read concurrently, drew repeated `Wasted call — file unchanged` rebukes and once a full API cancellation cascade. The damning part: by the final week the fix was written in Claude's own `feedback-small-tool-batches` memory and *still* got tripped, including an eight-call duplicate Read of the very file warning against duplicate batches. _Evidence:_ `2026-05-29_1845` (two batch cancellations), `2026-05-30_0942` (the "flaky console" delusion), `2026-05-31_1538` (~30-call batch + API cascade), `2026-05-31_1629` (write→read race), `2026-05-31_1730` (8 wasted Reads of the memory file itself). [claude]

3. **Edit-before-Read misfires — small, avoidable, recurring across the whole build.** Repeatedly Claude issued an `Edit` against a file it had only `cat`-ed via Bash (which doesn't register as a tracked Read) or hadn't read at all, eating a `File has not been read yet` / `String to replace not found` error then a remedial Read. Each costs one round-trip; they recurred in at least seven sessions. _Evidence:_ `2026-05-30_0848`, `2026-05-30_1249` (four times), `2026-05-30_1703`, `2026-05-31_1235`, `2026-05-31_1538` (three consecutive, against a *fabricated* file image), `2026-05-31_1813` (three times). [claude]

4. **Opus ran nearly everything, including overwhelmingly mechanical workloads — an unexamined model-fit lever on both sides.** Every leaf flags that the flagship model carried long stretches of checkbox-flipping, status-doc propagation, find-and-replace, and git plumbing that Sonnet (or Haiku) would handle at no quality cost. It is a soft fault — one interactive session can't switch mid-stream without friction, and continuity argues for one model — but it was *never raised by either party*, even in the session authoring a reviewer whose mandate is model-fit. _Evidence:_ `2026-05-30_2224` ("low-difficulty edit session ran entirely on Opus"), `2026-05-31_1235` ("Opus on all-prose, low-leverage workload"), `2026-05-31_1846`, `2026-05-31_1908` (ironic: unexamined while authoring the model-fit lens). [claude|human]

5. **Subagents and plan mode — the two features most fit for this build — went chronically unused, most glaringly on repetitive multi-unit loops.** The P5/P7 unit work was a textbook 15-iteration parallelizable job, and the course *teaches* subagents (U13) and worktrees (U16); yet the rollouts were scoped as serial main-thread work every time. Plan mode was skipped on review-gated spec builds — once causing a premature design commit that cost three remediation turns. _Evidence:_ `2026-05-30_0848` (16-unit loop never captured as a command/plan), `2026-05-30_2322` ("biggest missed leverage: U2–U16 rollout scoped serial"), `2026-05-31_0832` (serialized de-coding pass), `2026-05-31_1235` (plan mode missed, premature design commit), `2026-05-30_1557` (plan mode unused on multi-step authoring). [claude]

6. **The human's mid-session interventions were the highest-leverage economy moves in the corpus — but they were corrective, supplied *to* the collaboration rather than self-generated by Claude.** Repeatedly the human stopped a wrong trajectory at the cheapest possible moment, named a root cause, or injected missing context — and the fix held afterward. Several leaves note this is "economy imposed on the collaboration, not supplied by Claude," who wasn't self-gating. _Evidence:_ `2026-05-30_0942` (two-step "flaky console" diagnosis → memory), `2026-05-31_1629` (race-condition naming), `2026-05-31_1813` (context injection averting wrong refresh), `2026-05-31_1353` (three cheap interrupts preventing rework), `2026-05-30_1415` (caught the duplicated-narrative tax). 7. **Verification was treated as first-class and reproduce-don't-assert — a durable strength that earned its keep repeatedly.** Smoke-test-before-suite, fail-clean-then-pass-on-solution verifier loops, adversarial tests of verifier teeth, grep-with-expected-count, byte-stable diffs, stash-and-rerun to isolate inherited failures. This discipline caught real bugs in-flight (three bugs + a spec discrepancy in `2026-05-30_0725` alone) and is the most consistently excellent economy behavior across the corpus. _Evidence:_ `2026-05-30_0725`, `2026-05-30_0816` (seeded-bug reproduction incl. `chmod 555`), `2026-05-30_1415` (behavior-equivalence verifier, adversarially tested), `2026-05-31_1813` (stash to prove R8 pre-existing). The hand-maintained JSON twin (no tool consumes it) recurred as dead-weight double-edits. _Evidence:_ `2026-05-30_1557` (two full rebuild cycles, `/new-unit` dogfood theater), `2026-05-31_1235` (premature design commit → three remediation turns), `2026-05-30_1703` / `2026-05-30_2224` (JSON twin hand-maintained, diagnosed but generator never proposed). ## Per party — recurring strengths & failure modes

**Human author.** Strengths were remarkably consistent and *improved*: textbook per-change commit/push gate discipline (never standing permission) in essentially every session; surgical, cheaply-placed interventions that diagnosed root causes and redirected work before it was built; and a maturing instinct to convert one-off corrections into durable fixes (memory files, CLAUDE.md sharpenings, the `check-evaluations` gate). The token-load probe in `2026-05-31_1908` is the apex — forcing a measured cost analysis is exactly this lens applied by the operator. The recurring failure mode is singular and shared: **never exercising the model-fit lever** — driving long mechanical sessions on Opus without down-shifting — and rarely nudging toward subagent/plan-mode delegation, leaving that leverage dormant. Secondary: a couple of interventions arrived *after* commits (reactive Socratic review in `2026-05-30_1557`) where an up-front framing would have collapsed three commits into one.

**Claude.** Strengths: genuinely fluent capability *selection* that grew over the build — AskUserQuestion at real forks, live-CLI verification over memory, subagent for one version lookup, right-sized tool architecture, and a verification-first reflex that caught real defects cheaply. It was consistently honest about its own waste and could be argued out of its own positions on sound grounds. Failure modes were equally consistent and stubborn: **duplicate-batch / write-read-race waste** that relapsed even with the fix in its own memory; **Edit-before-Read misfires** across the whole build; **serialized independent calls** (Reads, TaskCreates, Edits) where batching was free; and **self-gating that never materialized** — the human supplied the approval pauses and economy corrections Claude should have generated. The biggest dormant leverage was structural: never capturing the repetitive unit loop as a subagent fan-out or slash command despite authoring the very course that teaches it.

## Bottom line

Through my lens this collaboration was **fluent in capability, plateaued in economy**. Claude chose the right tool at the right fork with growing sophistication, treated verification as first-class, and built genuinely well-architected instruments — but its turn-by-turn economy never converged: the same duplicate-batch, race-condition, and Edit-before-Read waste that appeared in session one was still firing in the final week, with the corrective lesson sitting unused in its own memory. Model-fit and subagent/plan-mode delegation were leverage left on the table from start to finish, unexamined by either party. The human is the reason the economy held as well as it did — improving steadily into the session's economy regulator, diagnosing root causes and injecting context at the cheapest possible moments. The defining verdict: Claude used Claude Code's *features* close to their potential, but used its *turns* and *model* economically only when the human made it.
