## What worked / what didn't (both parties)

**Claude — strengths.** Priming was scoped and paginated, not a cold start or a context dump. Every version-specific value was sourced from the live CLI or official docs with provenance, honoring the project's no-memory rule with unusual fidelity. State was written down at every boundary — task boxes, decision entries, refresh steps, and the session capture — so the next session can read files rather than reconstruct from chat. Self-verification was differential and cheap (`git stash` to isolate a pre-existing PEND; byte-stability diffs; synthetic `added_in` tests), which keeps context clean of noise. Communication summaries were terse and structured.

**Claude — weaknesses.** A handful of independent tool calls serialized where a single block would have been more efficient (the redirect-refetch WebFetch chain; some conventions-gathering Reads). This never bit because the session ran at ~38% context, but the batching discipline was the one area below the bar the rest of the session set. (The 8.4 unreviewed commit is a real process miss, but it is a permission/approval failure, not a context-management one — out of this lens.)

**Human — strengths.** Opened with the repo's own priming skill; drove the build one reviewable slice at a time; forced pause-and-present checkpoints that kept the window from filling with unreviewed work; asked file-grounded questions and got file-grounded answers. The "are your memories stale?" probe was a genuinely sharp context-engineering hypothesis, and the demand for a file:line answer pushed Claude off recollection and onto the filesystem.

**Human — weaknesses.** Little to fault on this lens. The one tightening: when surfacing "what's new" *in* the learner page (a deviation from the approved design), the author relied on Claude to notice the design-gate implication — which it did — but a more explicit "does this need a design touch-up?" would have been even cleaner.
